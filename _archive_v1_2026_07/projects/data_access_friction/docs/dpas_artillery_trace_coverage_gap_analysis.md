# DPAS-to-Artillery Trace Coverage: Gap Analysis

Data Date: 2026-04-01 | Author: Shruti Tyagi


## **Data Sources**

This analysis links two tables to measure how many DPAS authorization requests can be traced back to their corresponding artillery traces.

**Table 1: fct_agent_execution_dpas_requests** — DPAS authorization requests made by AI agents.

- Each row represents one distinct permission check, deduplicated by COALESCE(artillery_trace_id, fbtraceid)
- The artillery_trace_id field is a composite value: the first 11 characters are the root trace ID generated by artillery, and the remaining characters identify the specific sub-span within that trace.
- fbtraceid is Meta's general request tracing ID.
**Table 2: fct_agent_security_traces **— Agent execution traces materialized by the security pipeline.

- Each row represents one traced agent execution along with related metadata (security events, user info, tool calls, etc.).
- The trace_id field contains the 11-character root trace ID.

**How they join:**

SUBSTR(artillery_trace_id, 1, 11) from Table 1 → matches trace_id in Table 2.

- Records with a NULL artillery_trace_id in Table 1 cannot be joined at all — they have no artillery trace to match against.
## **Deduplication Methodology**

This analysis uses deduplicated records to count distinct traces rather than individual permission checks from DPAS:

- Records with artillery_trace_id: Deduplicated by SUBSTR(artillery_trace_id, 1, 11) — if multiple DPAS records share the same first 11 characters, they represent the same underlying trace and are counted once. The latest record (by event_time) is kept as the representative.
- Records with NULL artillery_trace_id: Deduplicated by fbtraceid.
Why deduplicate? A single agent trace can generate many DPAS authorization requests. Deduplication gives us the count of distinct traces rather than individual permission checks, providing a more accurate picture of coverage gaps.

**Totals: **3,502,759 distinct trace records on April 1, 2026, of which 205,716 (**5.9%**) have an artillery_trace_id and 3,297,043 (**94.1%**) **do not.**

## **The Question**

Every time an AI agent requests data access, DPAS logs it. Separately, artillery traces capture the full execution of that agent session. Ideally, we'd be able to link every DPAS request back to its artillery trace — this would let us see exactly which agent sessions are accessing what data.

**The problem: **When we attempt this join,*** only 4.5% of distinct DPAS records successfully match*** to a trace in the security pipeline. The remaining 95.5% are unlinked — we know the data access happened, but we can't connect it to the agent session that triggered it.

This document walks through the matching funnel step by step to identify where records are lost and where the gap could be.

## **Step 0: Starting Point**

We have 3,502,759 distinct DPAS trace records on April 1, 2026. Each represents a distinct agent trace requesting data permission. 

**Our goal: **link as many as possible to their trace in the security pipeline.

## **Step 1: Does the DPAS record even have an artillery_trace_id?**

The join requires artillery_trace_id — but the vast majority of records don't have one.


|  | Count | % of Total |
| --- | --- | --- |
| Has artillery_trace_id | 205,716 | 5.9% |
| artillery_trace_id IS NULL | 3,297,043 | **94.1%** |


### GAP #1:** **3.3 million records (94.1%) are dead on arrival

They can never match because there's no join key.

Why does this happen? Each DPAS record has a data_source field that tells us how it was logged. There are two sources:

- dpas_accessmate — logged via AccessMate, which has better artillery integration
- dpas_others — logged outside AccessMate (direct DPAS calls that bypass the AccessMate instrumentation path)
Nearly all NULL-artillery records come from dpas_others:


| data_source | % of NULL-artillery records | % of records WITH artillery |
| --- | --- | --- |
| `dpas_others` | **98.6%** | 95.2% |
| `dpas_accessmate` | 1.4% | 4.8% |


**The takeaway:** dpas_others is the dominant source for both groups, but records without artillery are even more concentrated in it. The dpas_accessmate path is 3.4× more likely to carry artillery context (4.8% vs 1.4%), but it still only accounts for a small fraction of total volume. The core problem is that most DPAS calls potentially go through code paths that never interact with artillery at all.

## **Step 2: For records WITH artillery_trace_id — do they match security traces?**

We now look at just the 205,716 records that DO have artillery_trace_id and match against security traces.


|  | Count | % of 205.7K |
| --- | --- | --- |
| Matched in security (same day) | 157,520 | 76.6% |
| No match found | 48,196 | **23.4%** |


### **GAP #2:** 48K records (23.4%) have an artillery ID but can't be found in security traces

Why does this happen? 

### **A: Partition boundary (trace logged on a different day)**

We checked if these unmatched trace keys exist in the previous day (Mar 31) or next day (Apr 2) of security data.


| Check | Found? |
| --- | --- |
| Found in Mar 31 security traces | 131 (0.3%) |
| Found in Apr 2 security traces | 0 |
| **Still missing after checking 3 days** | **48,065 (99.7%)** |


**Verdict: **Partition boundaries explain only 0.3%. This is NOT the main cause.

### **B: Traces never reached artillery in the first place**

For these **48,065** traces, the DPAS code path logs an artillery_trace_id but the corresponding trace is not logged in the artillery logger. 


**Next step:** Investigate if this is a logging issue with artillery client silently failing or the agent call finishing before the trace is flushed or the records were potentially sampled in artillery.


## **Step 3: Bonus signal — FAILED_BY_SERVER is a diagnostic canary**

This gap is small in volume but diagnostic. FAILED_BY_SERVER is a rare event — only ~71,097 out of 3.5M DPAS records (2.0%). But among those 71K records, the NULL artillery rate is 100%:


| Auth Response | Total Records | % of All DPAS | % with NULL artillery_trace_id |
| --- | --- | --- | --- |
| SUCCESS | 3,340,652 | 95.4% | 94.0% |
| DENIED | 75,862 | 2.2% | 97.4% |
| **FAILED_BY_SERVER** | **71,097** | **2.0%** | **100.0%** |
| FAILED_BY_USER | 14,792 | 0.4% | 87.7% |


Why this matters — it's a code-path indicator, not a volume problem:

FAILED_BY_SERVER shows a perfect 100% null artillery rate. Every single FAILED_BY_SERVER trace lacks artillery context. This points to a potential bug in the error-handling code path — timeouts, retries, or fallback logic that drops trace headers before they can be logged.

This bug may also contribute to Gap #1: if the same code path that handles retries/errors is also used for normal requests in some agents, it could explain broader artillery propagation failures.


**Action:** Investigate the DPAS server error-handling path — artillery context may be lost during retry logic, timeout handling, or fallback code paths. Even though the direct impact is small (71K), fixing the underlying code path may improve instrumentation more broadly.

## **Step 5: Agent Name Mismatch Between DPAS and Security Traces**

Even among the 157,520 records that successfully match on trace_id, there is an agent name discrepancy between the two tables:


| Match Status | Record Count | % of Matched |
| --- | --- | --- |
| Agent name mismatch | 76,914 | 48.8% |
| Exact agent name match | 76,719 | 48.7% |
| Matches platform only | 3,887 | 2.5% |


Exact matches are nearly equal to mismatches (48.7% vs 48.8%).

### **Key Mismatch Patterns**

- **Pattern A:** Naming convention differences (hyphen vs underscore)
  - **The largest single mismatch: **analytics-agent in DPAS maps to analytics_agent in security traces. This is a systematic formatting inconsistency — the underlying agent is the same, but the naming convention differs between the two logging systems.
- **Pattern B: **DPAS logs the top-level agent; security logs the sub-agent
  - When a parent agent delegates to sub-agents, DPAS records the parent's name while security traces record whichever sub-agent actually executed. Examples:
  - claude_code (DPAS) → analytics_agent (security)
  - claude_code (DPAS) → datamate (security)
  - PlugboardGenClient (DPAS) → analytics_agent (security)
  - datamate (DPAS) → datamate-daiquery / datamate_aide_unidash (security)
- **Pattern C: **Claude Code instance fragmentation
  - Security traces log individual Claude Code instances with unique suffixes (e.g., claude_code_a09a1c0f623fe2b10) rather than the generic claude_code name used by DPAS. This creates dozens of low-count "mismatch" rows that are actually the same agent.
### **Impact**

Agent name mismatch does NOT affect join completeness (trace_id matching works independently of agent names), but it means any downstream analysis that groups by agent_name or tries to reconcile per-agent metrics across DPAS and security will see inconsistencies. Standardizing agent naming or maintaining a mapping table between the two systems would improve analytical consistency.


**Action: **Create a canonical agent name mapping table, or standardize the agent_name field across DPAS and security trace pipelines to use consistent naming conventions.
